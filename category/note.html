Chapter17 Smells and Heuristics

这一章，是前面描述的内容的总结，列举了一些“不简洁代码”的表现。简单记录一下。

  • 构建工程，应当只需要一条命令；测试工程，也应该只需要一条命令。
  • 注释掉的代码，没有被调用的代码，不用担心，直接删掉就行。版本控制器会帮你管理。
  • 最小惊讶原则：你实现的函数，应当具有其他程序员凭直觉认定的，你这个函数应当具有的功能。比如StringToDay函数，应当将字符串"Monday"转化成enum Day类型的对象，并且，支持缩写（"Mon"）、不缺分大小写（"monday"）。如果没有实现某些感觉上显而易见的功能，使用该函数的程序员就需要去看代码或文档。
  • 最重要的原则之一：去重复。不单单是代码的重复，还包括
    1.大段的if/else或switch/case结构，用多态和抽象工场代替
    2.相似的算法，用模板方法模式或策略模式代替

To write clean code, you must first write dirty code and then clean it.

Chapter8 Boundaries

边界指的是第三方库和应用程序的边界。

用包裹类，或适配器模式，来封装第三方代码（third-party code），可以：

  1. 限制第三方代码的功能
  2. 提供更一致和好用的接口
  3. 减少受第三方代码改变带来的影响

经常出现匪夷所思的bug，可能是由于第三方库导致的。所以，在使用之前应该做一个Learning Test（一遍学习文档，一遍写单元测试），确保我们需要的功能它都能正确提供。Learning Test好处：

  1. 帮助学习文档
  2. 确保得到如期的结果，避免以后bug不知道从哪儿来
  3. 第三方库版本更新后，继续用它测试是否符合预期（很多时候我们都不及时更新第三方库，因为我们怕更新后带来新的问题)

Chapter9 Unit Tests

  • TDD（Test Driven Development）的三法则：写了失败的UT再写成产代码；一旦可以保证生产代码失败，就停止写UT；一旦生产代码通过了UT，就停止写生产代码。这样一轮循环大约30s。

Chapter2 单机存储系统

2.1硬件基础

CPU架构

  • SMP（对称多处理）:经典的CPU架构，一个计算机上汇集了一组CPU（一个CPU往往包括不止一个核），对称工作、无主次或从属关系，共享相同的物理内存和总线。
    某些CPU还通过超线程技术（Hyper-Threading Technology），在一个核心上同时执行两个线程。
  • NUMA（非一致存储访问）:克服SMP中多CPU对总线资源的竞争，一个NUMA包括多个NUMA节点，一个NUMA节点就是一个SMP结构（n个CPU，1个内存）

IO总线

Intel的主板，一般为南、北桥架构

  • 北桥芯片：CPU，内存，高端SSD (高速IO)
  • 南桥芯片：网卡，硬盘，中低端SSD

网路拓扑

数据中心网络拓扑结构，主要分为

  • 三层结构：核心层、汇聚层、接入层。一个接入层交换机下的服务器部署在一个机架内
  • 三级CLOS网络：Google设计的扁平化拓扑结构（从wiki上看图，感觉挺复杂的）

Chapter1 概述

分布式存储特征

  • 可扩展：随着集群规模增大，整体性能表现为线性增长
  • 低成本：构建在普通PC上，实现自动容错、自动负载均衡
  • 高性能
  • 易用：提供易用的外部接口，也具备完善的监控和运维工具

鸡汤：

一般来讲，工程师如果能够深入理解分布式存储系统，理解其他互联网后台架构不再会有任何困难。

分布式存储分类

  • 非结构化数据：文档、图片、视频等
  • 结构化数据：数据库中的表
  • 半结构化数据：HTML文档，模式结构和内容混在一起
