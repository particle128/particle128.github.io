Chapter8 Boundaries

边界指的是第三方库和应用程序的边界。

用包裹类，或适配器模式，来封装第三方代码（third-party code），可以：

  1. 限制第三方代码的功能
  2. 提供更一致和好用的接口
  3. 减少受第三方代码改变带来的影响

经常出现匪夷所思的bug，可能是由于第三方库导致的。所以，在使用之前应该做一个Learning Test（一遍学习文档，一遍写单元测试），确保我们需要的功能它都能正确提供。Learning Test好处：

  1. 帮助学习文档
  2. 确保得到如期的结果，避免以后bug不知道从哪儿来
  3. 第三方库版本更新后，继续用它测试是否符合预期（很多时候我们都不及时更新第三方库，因为我们怕更新后带来新的问题)

Chapter9 Unit Tests

  • TDD（Test Driven Development）的三法则：写了失败的UT再写成产代码；一旦可以保证生产代码失败，就停止写UT；一旦生产代码通过了UT，就停止写生产代码。这样一轮循环大约30s。
  • 测试代码和生产代码一样重要，也要简洁、可读。因为如果一开始测试代码标准不高，随着生产代码的演化，阅读和修改单元测试的成本会非常高，高到一定程度 ...

Chapter5 Formatting

  • 你的代码可能随着版本的升级，逐渐被替换；但是你的代码风格/格式会被后继开发者一直传承下去。
  • 文件的长度：通常200行以内，最多500行
  • C++类文件，通常遵循scissors rule，即上面是成员函数，下面是成员变量；java相反，上面是成员变量，下面是共有函数
    根据Sun的Code Convention for Java，类中的成员的顺序如下：
    1.static成员变量：public -> protected -> package -> private
    2.non-static成员变量：public -> protected -> package -> private
    3.构造函数
    4.成员函数：按照语义顺序，而不是访问级别

Chapter2 Meaningful Names

  • 使用表明意义（intention-revealing）的名字，而不是简单的名字+注释。比如elapsedTimeInDays
